# Заметка: День 1 — SkipList (HLR)

## 1) Что реализовано

Пакет `internal/skiplist` — in-memory хранилище на основе SkipList.

Поддерживаемые операции:
- `Put(key, value []byte)` — вставка и обновление
- `Get(key []byte)` — поиск по ключу
- `Delete(key []byte)` — удаление
- `Scan(start, end []byte)` — ленивый итератор по диапазону `[start, end)`

## 2) Инварианты и семантика

**Обновление ключа.** Повторный `Put` с тем же ключом перезаписывает значение — в списке не возникает дубликатов. Новый узел не создаётся.

**Scan([start, end)).** `end` — exclusive-граница: узел с ключом `== end` не включается в результат. `nil` трактуется как −беск. (start) или +беск. (end). Итератор ленивый — не копирует данные в массив, обходит по указателям уровня 0.

**Отсутствующие ключи.** `Get` и `Delete` возвращают `ErrNotFound`. Паник нет.

**Удаление.** После `Delete` узел полностью исключается со всех уровней. Если верхние уровни опустели — `s.level` уменьшается.

## 3) Принятые решения

`maxLevel` = 32 Покрывает до ~4 млрд элементов при p=0.5; проще, чем вычислять из N;
`p` = 0.5 | Классический coin-flip: O(log₂ N) ожидаемая высота, сбалансированное потребление памяти;
Генерация уровня `rng.Float64() < p` (coin-flip);
`New(seed int64)` Детерминированные тесты: при одинаковом seed список всегда строится одинаково, ошибки воспроизводимы |

## 4) Контроль (как проверялось)

**Тесты** (`make test-day1`):
- `TestPut_UpdateExistingKey` — обновление перезаписывает, не дублирует
- `TestDelete_NonExistentKey` — ErrNotFound при удалении несуществующего
- `TestScan_ExclusiveEndBoundary` — end-граница не включается
- `TestPut_EmptyKey` — пустой ключ работает (лексикографически наименьший)

**Бенчмарк `BenchmarkGet` — тренд при росте N** (Apple M1):

```
N=100       164 ns/op    24 B/op    2 allocs/op
N=1000      196 ns/op    29 B/op    2 allocs/op
N=100000    268 ns/op    31 B/op    2 allocs/op
```

`Get` растёт от 164 до 268 ns при N ×1000 — подтверждает O(log N): при росте N в 1000 раз время растёт лишь в 1.6 раза.

## 5) Известные ограничения

- Не thread-safe: курс предполагает однопоточность.
- Нет снапшотов или MVCC: нельзя итерироваться и одновременно изменять список.
- `Scan` возвращает копии — 2 аллокации на элемент (подробнее в разделе 5).
- `maxLevel=32` — фиксированная константа; при очень малых N (< 10) несёт избыточный overhead в sentinel head (32 nil-указателя вместо, например, 4).
