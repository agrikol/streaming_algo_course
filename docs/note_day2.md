## 2) Семантика

**Обновление ключа** — повторный `Put` с тем же ключом перезаписывает значение. В Memtable новое значение вытесняет старое (SkipList хранит последнее). В SSTables может существовать несколько версий одного ключа — побеждает та, что в более новом файле.

**`Scan([start, end))`** — возвращает ключи в порядке возрастания, включая `start`, исключая `end`. Реализован через merge-итератор поверх Memtable и всех SSTables (от нового к старому). При одинаковых ключах из нескольких источников побеждает самый новый. Tombstone-записи не возвращаются пользователю.

**Отсутствующие ключи** — `Get` возвращает `ErrNotFound`. 
**Удалённые ключи** — `Delete` записывает tombstone (маркер `tagDelete = 0x02`) в Memtable и WAL. При `Get` или `Scan` tombstone скрывает ключ от пользователя. Физическое удаление происходит только при compaction.


## WAL

Формат: length-prefixed записи.
```
[4 байта uint32 LE] totalPayloadLen
[1 байт]            opType (1=Put, 2=Delete)
[4 байта uint32 LE] keyLen
[keyLen байт]       key
[4 байта uint32 LE] valLen  (0 для Delete)
[valLen байт]       value
```

- При Open открывается с O_APPEND — все новые записи в конец, sequential I/O.
- Перед открытием на запись — replayWAL проигрывает существующий WAL в Memtable. Это и есть crash recovery.
- После flush Memtable WAL удаляется и создаётся пустым. Логично: данные из Memtable теперь в SSTable, журнал больше не нужен.
- При корректном Close Memtable сбрасывается и WAL удаляется -> следующий Open стартует с чистого листа.

Почему length-prefixed: зная размер записи заранее, можно прочитать её одним куском (`io.ReadFull`). Если payload не дочитался (`io.ErrUnexpectedEOF`) — значит краш произошёл в середине записи, останавливаем replay. Это позволяет отличить "конец файла" от "битой хвостовой записи".

## SSTable

Структура файла:
```
[Секция данных]  — записи подряд: keyLen(4) + key + opType(1) + valLen(4) + value
[Sparse Index]   — одна запись каждые 16 записей данных: keyLen(4) + key + offset(8)
[Footer 16 байт] — indexOffset(8) + magic(8)
```

`indexStride = 16` — баланс между размером индекса и количеством записей, которые нужно пропустить при поиске. При 1 млн записей индекс содержит ~62 500 точек.

Footer хранит `indexOffset` — при открытии Reader читает последние 16 байт, находит индекс и загружает его в память. Магическое число `0x53535441424c4531` ("SSTABLE1") защищает от открытия чужого файла.

Add пишет запись в data-секцию через bufio.Writer и руками отслеживает curOffset. Каждые 16 записей в index добавляется индексная точка. Close сбрасывает индекс и footer

NewReader сразу при создании вызывает loadIndex — читает последние 16 байт (footer), оттуда indexOffset, затем читает индекс целиком в RAM. Дальше для поиска нужна только директория с одним open-fd.

Через бинарный поиск (sort.Search) по индексу находим последний блок, чей первый ключ ≤ start, и начинаем последовательное чтение оттуда

Для tombstones в SSTable используется поле `opType = 2` — tombstone хранится явно (необходимо при Scan, чтобы скрыть более старые версии ключа из других файлов).

## LSM

### Tombstones в Memtable

В SkipList нельзя хранить "метаданные" записи — только `[]byte`. Решение: тэгирование значения первым байтом:
- `0x01` + value = обычная запись (Put)
- `0x02` = tombstone (Delete)

---

- При чтении из Memtable тэг снимается. Это позволяет Memtable хранить tombstones без отдельной структуры данных.
- memBytes (lsm.go:78) грубо считает суммарный объём — нужен только чтобы решать, когда делать flush. Порог по умолчанию 4 МБ. Когда memBytes >= порог:
1. Создаём sst-NNNNNN.sst с новым seq.
2. Итерируемся по Memtable, по тэгу вызываем w.Add или w.AddTombstone.
3. w.Close() — пишет индекс и footer.
4. Открываем файл на чтение, добавляем sstableHandle в e.sstables.
5. Создаём новый пустой SkipList, обнуляем memBytes.
6. Удаляем WAL и создаём новый пустой.
7. Если файлов накопилось ≥ 4 → compact.

- Put / Delete
WAL — первым, Memtable — вторым. Так если процесс умрёт после WAL, но до Memtable — replay восстановит. Если умрёт после WAL и Memtable, но до flush — replay восстановит то же состояние, что было в Memtable.
- Get
Порядок поиска:
Memtable — самые свежие данные.
SSTables от новейшего к старейшему — e.sstables отсортированы по seq (старый → новый)
Tombstone в любом источнике → ErrNotFound. Это правило обязательно: иначе после Delete старое значение из старого SSTable всплыло бы.
В каждом SSTable создаётся Iterator(key, nil) и берётся первая запись — если она совпадает с key, это ответ. 

### Compaction

Стратегия "слить всё L0": при накоплении ≥ 4 SSTable файлов — сливаем все в один через k-way merge (min-heap). Tombstones при compaction удаляются физически, так как мы сливаем все уровни и более старых копий ключа нет.

### Close и WAL ротация

При `Close()` Memtable сбрасывается в SSTable, старый WAL удаляется. При следующем `Open()` данные восстанавливаются из SSTable (WAL пустой или отсутствует). При краше (без `Close`) — данные восстанавливаются из WAL через replay в Memtable.

После каждого flush WAL ротируется: старый файл удаляется, создаётся новый пустой.

## 4) Контроль

`TestLSMStore_PersistAcrossRestart`

Проверяет полный цикл: WAL-запись → flush → удаление WAL → открытие SSTable → чтение.

Крайние случаи (`make test-day2`):
- Пустое значение (`Put("k", "")`) — сохраняется и возвращается корректно.
- Пустой ключ (`Put("", v)`) — работает и переживает рестарт (проверка keyLen=0 в WAL/SSTable).
- Повторный Put — возвращается последнее значение (`v3`, не `v1`).
- Delete → Get возвращает `ErrNotFound`.
- Get несуществующего ключа → `ErrNotFound`.
- Scan `[b, d)` — возвращает `b`, `c`; не возвращает `a` и `d`.
- Scan — удалённые ключи не попадают в результат.
- Scan `[a, a)` (пустой диапазон) — пустой результат.

**Put (~1.8–2.1 мкс)** — Рост с N логарифмический и почти незаметен, потому что SkipList `O(log N)` быстрее дисковой задержки. 

**Get (~87–143 нс)** — только SkipList `O(log N)`, без I/O (данные помещаются в Memtable). Рост логарифмический: при 100× росте N время растёт лишь в 1.6×. 1 аллокация — копия возвращаемого значения.
